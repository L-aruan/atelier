## 1. 项目概述

### 1.1 产品定位

MediaBox 是一个面向电商运营、自媒体创作者、中小企业设计团队的**通用媒体工具聚合平台**。核心理念是「工具导航 + 封装层」——聚合互联网上已有的优秀工具能力，提供统一的中文化体验和增值功能（工作流串联、批量处理、文件管理）。

### 1.2 核心价值（优先级排序）

1. **体验和本地化**：中文界面、快速访问、流畅的交互体验
2. **工作流串联**：多工具串成流水线，自动化重复操作
3. **批量处理**：大批量文件处理，突破单工具网站的数量限制
4. **统一文件管理**：处理历史、结果审查、一键导出

### 1.3 目标用户

- 电商运营人员（商品图批量处理）
- 自媒体/内容创作者（视频剪辑、封面设计）
- 中小企业市场/设计团队（日常素材加工）

### 1.4 工具类别

| 类别 | 示例工具 |
|------|---------|
| 图片处理 | 裁剪、压缩、去背景、加水印、格式转换、批量改尺寸 |
| 视频处理 | 剪辑、转码、压缩、加字幕、提取音频、去水印 |
| 设计/排版 | 海报生成、封面制作、模板套用、拼图 |
| 音频处理 | 降噪、格式转换、剪辑、语音转文字 |
| 文档/数据 | PDF 处理、Excel 批量操作、OCR 文字识别 |
| AI 增强 | AI 抠图、AI 扩图、AI 写文案、AI 生成图片 |

### 1.5 AI Key 策略

- 用户可配置自己的 API Key，使用高质量模型
- 平台提供通用 Key（低成本模型），零配置即可使用
- AI 网关统一路由，对工具层透明

## 2. 系统架构

### 2.1 混合架构概览

三层架构，轻量任务跑浏览器端，重任务发到后端 Serverless 函数。

```
┌─────────────────────────────────────────────────────┐
│                    用户浏览器                         │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │
│  │ 客户端处理引擎 │ │  工作流编排器  │ │  文件管理器   │ │
│  │ FFmpeg.wasm   │ │ 可视化流水线  │ │ IndexedDB    │ │
│  │ Squoosh WASM  │ │ 拖拽连线     │ │ 处理历史      │ │
│  │ PDF.js        │ │ 自动选执行端  │ │ 批量上传下载   │ │
│  │ Tesseract.js  │ │              │ │              │ │
│  └──────────────┘ └──────────────┘ └──────────────┘ │
│        轻量任务本地完成 · 重任务发到后端                  │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│                 Next.js API Layer                     │
│  ┌───────────┐ ┌──────────┐ ┌────────┐ ┌──────────┐│
│  │工具注册中心│ │ AI 网关   │ │任务调度 │ │用户/存储  ││
│  │ Manifest  │ │ Key 路由  │ │BullMQ  │ │PostgreSQL││
│  │ 版本管理   │ │ 模型选择  │ │进度推送 │ │S3/OSS    ││
│  └───────────┘ └──────────┘ └────────┘ └──────────┘│
└──────────────────────┬──────────────────────────────┘
                       │ 重处理任务
┌──────────────────────▼──────────────────────────────┐
│               Serverless Workers                     │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │
│  │  视频 Worker  │ │  AI Worker   │ │  文档 Worker  │ │
│  │ FFmpeg 转码   │ │ 抠图/扩图    │ │ PDF 处理     │ │
│  │ 字幕烧录     │ │ 文案生成     │ │ OCR 识别     │ │
│  │ 音频提取     │ │ 语音转文字   │ │ Excel 操作   │ │
│  └──────────────┘ └──────────────┘ └──────────────┘ │
│              按需启动，用完即停                        │
└─────────────────────────────────────────────────────┘
```

### 2.2 三层分工

| 层 | 职责 | 特点 |
|----|------|------|
| 浏览器层 | 图片裁剪/压缩/格式转换、轻量视频剪辑、PDF 预览等高频操作 | 即时完成，零服务器成本 |
| API 层 | 工具注册发现、AI Key 路由、任务调度、用户管理 | 薄 API 层，轻量部署 |
| Worker 层 | 视频转码、AI 推理、大文件处理 | 按需启动，用完即停 |

### 2.3 跨平台策略

- **Web 模式**：浏览器访问，天然跨平台
- **桌面模式**：Tauri 2.0 打包，Mac (.dmg) + Windows (.msi/.exe) 双平台
- 每个工具发布前必须通过 Mac + Windows 双平台测试
- 桌面版可选择打包单个或多个工具
- 离线可用的工具完全本地运行，需联网的工具标注提示

## 3. 工具系统设计

### 3.1 Tool Manifest

每个工具通过 manifest.json 声明自身能力：

```json
{
  "id": "image-crop",
  "name": "图片裁剪",
  "category": "image",
  "description": "自由裁剪、按比例裁剪、批量裁剪到指定尺寸",
  "icon": "crop",
  "version": "1.0.0",
  "runtime": {
    "client": true,
    "server": false,
    "offline": true,
    "downloadable": true
  },
  "input": {
    "accept": ["image/jpeg", "image/png", "image/webp"],
    "maxSize": "50MB",
    "batch": true
  },
  "output": {
    "formats": ["jpeg", "png", "webp"]
  },
  "engine": "@mediabox/engine-image",
  "component": "./ImageCropTool.tsx"
}
```

关键字段说明：
- `runtime.client`：是否可在浏览器端运行
- `runtime.server`：是否需要后端处理
- `runtime.offline`：是否支持离线使用
- `runtime.downloadable`：是否可打包为桌面应用
- `engine`：依赖的共享处理引擎
- `input.batch`：是否支持批量处理

### 3.2 工具接口标准

每个工具实现统一接口：

```typescript
interface MediaBoxTool {
  manifest: ToolManifest
  Component: React.FC<ToolProps>
  process(input: FileInput, options: ToolOptions): Promise<FileOutput>
}
```

工具开发者只需关注 UI 组件和处理逻辑，平台壳提供路由、布局、文件管理等基础设施。

### 3.4 工具来源策略

- **开源自建为主**：基础工具（裁剪、压缩、格式转换、转码等）使用成熟开源库实现，完全可控
- **第三方 API 补充**：专业 AI 能力（高质量抠图、AI 生图等）按需接入第三方 API
- 第三方工具在 manifest 中标记 `runtime.server: true` + `runtime.offline: false`

### 3.5 快速集成机制

后续新增工具的标准流程：

1. **CLI 脚手架**：`mediabox create-tool <name> --category <cat>` 生成标准骨架
2. **开发模式**：`mediabox dev` 启动本地开发服务器，热更新实时预览
3. **Manifest 校验**：自动检查 manifest 合法性和输入输出兼容性
4. **发布注册**：`mediabox publish` 注册到平台工具目录，支持版本管理

## 4. 工作流引擎

### 4.1 工作流模型

- 每个工作流是一个有向无环图（DAG），节点 = 工具，边 = 数据流转
- 相邻节点输入输出格式自动校验兼容性
- 支持分支（一个输入同时走多条线路）

### 4.2 双模式编辑

| 模式 | 交互方式 | 适用场景 |
|------|---------|---------|
| 简单模式（默认） | 步骤列表，从上往下排，每步选工具+参数 | 线性流水线，大多数用户 |
| 高级模式（可选） | 画布节点编辑器（React Flow），拖拽连线 | 分支、并行等复杂逻辑 |

### 4.3 智能执行调度

- 引擎根据节点 manifest 自动决定执行位置
  - `runtime.client: true` → 浏览器端执行
  - `runtime.server: true` → 发到后端 Worker
- 相邻客户端节点数据内存传递，零网络延迟
- 服务端节点自动上传 → 处理 → 下载结果 → 继续下一步
- 批量模式并行处理，实时进度推送

### 4.4 工作流模板

预设常用模板：
- 电商商品图批处理（去背景 → 裁剪 → 加水印 → 压缩）
- 自媒体封面制作（裁剪 → 加文字 → 导出多尺寸）
- 视频发布流水线（转码 → 加字幕 → 压缩 → 导出）

用户可保存自定义工作流，下次一键复用。

## 5. UX 设计

### 5.1 核心交互原则

- **双向动线**：支持「工具优先」和「文件优先」两种入口
- **预览确认**：批量处理前先跑 2-3 张预览，确认效果后再全量执行
- **结果可审查**：批量完成后进入审查视图，问题项可单独重试
- **常用置顶**：钉选工具 + 最近使用，减少重复寻找

### 5.2 首页布局

```

│  Logo   工具库 | 工作流 | 我的文件    🔍搜索  登录 │ ← 顶部导航

│   ┌──────────────────────────────────────┐     │
```

### 5.3 文件优先流程

1. 用户拖文件到首页拖放区
2. 平台识别文件类型，智能推荐可用工具
3. 用户选择工具 → 直接进入操作界面（文件已就绪）
4. 或选择「创建工作流」串联多个工具

### 5.4 批量处理流程

1. 上传多个文件 → 设置参数
2. 先处理 2-3 张预览 → 用户确认效果
3. 确认 → 全量处理，实时进度条
4. 完成 → 结果审查视图（缩略图网格）
5. 成功项可直接下载，问题项可单独调参重试
6. 支持原图/处理后左右对比

### 5.5 工具操作页布局

- 左侧：文件上传/拖入区 + 批量文件列表
- 中间：工具主操作区（裁剪框、参数调节等），实时预览
- 右侧：输出预览 + 下载

## 6. 技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 前端框架 | Next.js 14 (App Router) + React 18 | SSR 首屏、流式渲染 |
| UI | Tailwind CSS + Radix UI | 快速出样式 + 无障碍交互 |
| 状态管理 | Zustand | 轻量、工具状态隔离 |
| 客户端处理 | FFmpeg.wasm / Squoosh / PDF.js / Tesseract.js | 浏览器端重处理 |
| 工作流 | 自研步骤列表 + React Flow（高级） | 双模式编排 |
| 后端 API | Next.js API Routes + tRPC | 全栈类型安全 |
| 任务队列 | BullMQ (Redis) | 异步重任务 |
| 数据库 | PostgreSQL + Prisma ORM | 用户/工具/工作流元数据 |
| 对象存储 | S3 兼容（阿里云 OSS / MinIO） | 用户文件 |
| AI 网关 | 自建路由层 | 多 key、多模型路由 |
| 桌面打包 | Tauri 2.0 | Mac + Windows |
| Monorepo | pnpm workspaces + Turborepo | 多包管理 |
| CI/CD | GitHub Actions | 测试 + 构建 + 部署 |

## 9. 协作开发指南

### 9.1 分工建议

| 角色 | 职责范围 |
|------|---------|
| 前端开发 | 平台壳 UI、工具 UI 组件、批量/工作流交互 |
| 引擎开发 | WASM 引擎封装、工作流执行引擎、批量处理引擎 |
| 后端开发 | API Routes、AI 网关、任务队列、用户系统 |
| 工具开发 | 各具体工具的 manifest + UI + 处理逻辑 |

### 9.2 开发规范

- **分支策略**：main（生产）→ develop（开发）→ feature/tool-xxx（工具分支）
- **PR 规范**：每个工具/功能一个 PR，需通过 CI + Code Review
- **工具独立性**：每个工具包可独立开发、测试、发布，不影响其他工具
- **类型安全**：全链路 TypeScript，tRPC 保证前后端类型一致

### 9.3 并行开发策略

Monorepo 的优势在于多人可同时开发不同模块而互不阻塞：

```
开发者 A: packages/platform/web     (平台壳)
开发者 B: packages/engines/image    (图片引擎)
开发者 C: packages/tools/image-crop (裁剪工具)
开发者 D: packages/tools/image-compress (压缩工具)
```

只要接口（shared/types）先定义好，各模块可完全并行推进。
